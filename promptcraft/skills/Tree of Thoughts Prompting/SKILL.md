# 思维树提示 Skill

## 任务

根据用户提供的任务、待处理内容、输出格式和约束，生成一个使用 Tree of Thoughts（ToT，思维树）的 Prompt。默认目标是“撰写提示词”，不是直接完成用户任务；只有当用户明确要求执行生成后的 Prompt 时，才继续产出最终结果。

ToT 的核心不是让模型“多想一点”，而是把问题求解视为对多个候选思路的搜索：每个节点是一个可公开的中间思路或部分解；模型在每一步生成多个候选、评价候选、保留最有希望的分支、必要时剪枝或回溯，最后用最优路径或综合结果给出答案。

## 对用户给出的两个示例的判断

“Imagine three different experts...” 和 “Three experts with exceptional logical thinking skills...” 可以体现 ToT 的一部分思想：多候选、多人视角、逐轮共享、发现错误后退出、迭代改进。它们适合作为直观的“专家协作式思维树”简化提示。

但它们还不等于完整 ToT。缺少的关键控制包括：

- 明确定义每个 `thought` 是什么粒度：一句策略、一个公式、一个计划段落、一个部分答案等。
- 限制分支数和最大深度，避免输出失控。
- 明确每轮如何评价候选：分数、等级、投票、可行/不可行、风险。
- 明确如何保留、剪枝、回溯或合并分支。
- 明确最终决策规则：最高分路径、投票获胜、综合多个优秀分支，或在冲突时使用评估标准裁决。
- 明确输出格式，尤其是表格字段和最终答案位置。

所以，本 skill 生成的提示词应保留“专家协作”的可读性，同时补上 ToT 的搜索控制结构。

## 输入要求

- 用户任务：说明希望模型完成什么任务。
- 待处理内容：需要推理、规划、选择、创作、排错或生成的具体问题；如果暂未提供，保留占位符。
- 输出格式：说明最终答案的结构、字段、长度、语言、代码格式或是否只输出最终答案。
- 可选案例：用户可以在第一次请求中直接提供 0~5 组 `任务 -> 候选分支/专家思路 -> 评估/剪枝依据 -> 最终合并结果`，用于展示分支生成、评估和合并风格。
- 可选信息：专家角色、分支数、最大深度、搜索策略、评价标准、剪枝规则、是否展示树过程。

示例不是必需项。若用户已经附上案例，直接吸收案例中的分支生成方式、评估标准、剪枝口径和最终合并风格；若未检测到案例，不要追问或中断流程，按零样本思维树提示生成，并在生成结果中简短说明“未检测到用户案例，因此按零样本思维树提示生成”。

## 流程概览

1. 判断任务是否适合 ToT：复杂规划、开放式创作、多方案决策、数学/符号搜索、代码排错、策略选择、约束满足类任务更适合；简单问答、单步抽取、短文本改写通常不需要。
2. 读取 `assets/template.md`，默认生成“结构化单轮 ToT Prompt”。
3. 如果用户在首次请求中提供案例，整理为可选参考案例；如果没有案例，写入零样本说明。
4. 根据任务选择思维单元粒度：
   - 数学/符号任务：一步操作、一个中间式、一个局部解。
   - 写作/创意任务：一个写作计划、一个段落方案、一个叙事方向。
   - 决策/规划任务：一个策略、一个行动方案、一个风险处理路径。
   - 代码/排错任务：一个假设、一个诊断步骤、一个修复方案。
5. 配置搜索参数：默认分支数 `3`、最大深度 `3`、每轮保留 `2` 个分支；用户未指定时使用保守值。
6. 明确评价与剪枝规则：按正确性、可行性、约束匹配、风险、成本、清晰度等标准评分或投票。
7. 明确最终合并方式：选择最高分路径，或综合多个优秀分支后给出一个最终答案。
8. 返回生成的 Prompt 给用户；不要默认执行 Prompt。

## Prompt 设计规则

- `thought` 是可公开的中间语义单元，不是隐藏思维链。要求模型输出“候选思路/方案摘要/评估理由”，不要要求暴露完整内部推理。
- 每轮必须先生成多个候选，再评价，再保留/剪枝；不要一条路走到底。
- 输出必须有明确状态表：轮次、分支编号、候选思路、评分/判断、保留或剪枝原因。
- 分支数和深度必须有限制，防止 ToT 提示词变成无限展开。
- 专家角色可用于增强多样性，但专家不是 ToT 的必要条件；关键是分支搜索与状态评价。
- 如果任务有硬约束，评价标准必须把硬约束放在第一位。
- 如果所有分支都低质量，允许回溯到上一轮重新生成候选。
- 在输出评估与剪枝过程时，请保持精简。对于明显违背物理常识或分布式原理的劣质分支，可直接在内部思维链中抛弃，仅输出具备高可行性的候选分支进行对比。
- 对严格格式输出任务，树搜索过程可以内部完成，最终只输出用户指定格式。

## 搜索策略选择

- `beam` / 宽度优先近似：每轮生成多个候选，保留前 `b` 个继续展开。适合规划、创作、数学搜索和多数通用任务。
- `dfs` / 深度优先近似：优先深入最高分分支，遇到不可行或低分时回溯。适合谜题、约束满足、排错。
- `expert-panel` / 专家协作：让 3 个或更多专家各自产生候选，再相互评价和修正。适合用户喜欢可读讨论过程的场景，但仍要加入分支数、深度、评分和决策规则。

## 可复用脚本

```bash
python scripts/render_tree_of_thought_prompt.py input.json
python scripts/render_tree_of_thought_prompt.py input.json --style expert-panel
python scripts/render_tree_of_thought_prompt.py input.json --style compact
```

推荐 JSON 输入：

```json
{
  "task": "为复杂问题生成思维树提示词",
  "role": "你是一名严谨的多路径问题求解助手",
  "target_input": "{用户输入的问题}",
  "output_format": "先输出思维树过程表，再输出最终答案。",
  "examples": [
    {
      "task": "示例任务",
      "branches": ["候选路径 A", "候选路径 B", "候选路径 C"],
      "evaluation": "根据可行性、风险和目标一致性剪枝",
      "final": "整合最佳分支后形成最终提示词"
    }
  ],
  "constraints": ["每轮最多 3 个候选", "最多 3 轮", "最终答案必须简洁"],
  "branch_count": 3,
  "max_depth": 3,
  "keep_count": 2,
  "search_strategy": "beam",
  "evaluation_criteria": ["正确性", "可行性", "约束匹配", "风险"],
  "decision_rule": "选择最高分路径；若两个分支互补，则综合后给出一个答案。"
}
```

`examples` 可省略；省略时按零样本思维树提示生成，并在案例区说明原因。

## 注意事项

- ToT 成本通常高于 CoT，因为它要生成和评价多个分支；只有复杂任务值得使用。
- 用户给出的两个专家示例可以作为友好入口，但建议补上表格化状态、分支限制、评分标准和最终决策方式。
- 不要把“专家人数”误当成 ToT 的核心。核心是：候选生成、状态评价、搜索/剪枝、最终选择。
- 用户提供案例时，可以作为少样本参考嵌入提示词，但案例服务于分支生成、评估、剪枝和合并风格，不应削弱思维树结构。
- 如果用户只想要最终答案，生成的 Prompt 应要求模型内部执行 ToT，最终只交付目标格式。
