# 后退提示 Skill

## 任务

根据用户提供的任务、待处理内容、输出格式和可选案例，生成一个使用 Step-Back Prompting 的 Prompt。默认目标是“撰写提示词”，不是直接完成用户任务；只有当用户明确要求执行生成后的 Prompt 时，才继续产出最终结果。

Step-Back 的核心是：遇到复杂、多细节、容易被局部信息带偏的问题时，先后退一步，把原问题抽象成更高层的概念、原则、通用问题或事实范围；再把这个抽象层得到的原则或事实作为依据，回到原问题完成推理。

## 输入要求

- 用户任务：说明希望模型完成什么任务。
- 待处理内容：需要回答、分析、计算、判断或生成的具体问题；如果暂未提供，保留占位符。
- 输出格式：说明最终答案的结构、字段、长度、语言、代码格式或是否只输出最终答案。
- 可选案例：用户可以在第一次请求中直接提供 0~5 组 `原始问题 -> 后退问题 -> 抽象答案/原则 -> 最终答案`，用于展示如何后退抽象；用户不提供案例时使用零样本后退提示。
- 可选信息：角色设定、约束条件、是否展示后退过程、是否使用检索增强上下文。

示例不是必需项。若用户已经附上案例，直接吸收案例中的后退抽象方式和输出风格；若未检测到案例，不要追问或中断流程，按零样本后退提示生成，并在生成结果中简短说明“未检测到用户案例，因此按零样本后退提示生成”。

## 流程概览

1. 判断任务是否适合后退提示：科学/数学原则题、知识问答、时间或地点约束、多跳推理、政策/合同/长文分析、复杂决策更适合；简单改写、单步分类或纯创意任务通常不需要。
2. 默认生成”单轮后退提示”；如果外部流程支持多次调用模型，可拆为”抽象阶段”和”基于抽象推理阶段”。
3. 如果用户在首次请求中提供案例，把案例格式化到 Prompt 中；如果没有案例，明确说明按零样本后退提示生成，并要求模型自行学习“先抽象、再回到原题”的策略。
4. 在 Prompt 中明确两步：
   - Abstraction：先不要直接回答原题，生成高层后退问题，并提取相关概念、原则、背景事实或通用框架。
   - Abstraction-grounded Reasoning：基于抽象层内容回到原始细节，完成推理并输出最终答案。
5. 明确输出控制。如果用户要求只输出最终答案，则让模型内部完成后退过程；如果用户要求可解释过程，则展示“后退问题 / 抽象原则或事实 / 回到原题推理 / 最终答案”。
6. 返回生成的 Prompt 给用户；不要默认执行 Prompt。

## Prompt 设计规则

- 后退问题应比原始问题更抽象、更通用、更容易回答，但必须覆盖原题所需的信息。
- 不要把后退提示写成低层子问题拆解；后退提示是“上升到概念层”，不是“拆成步骤清单”。
- 抽象层可以是领域原则、公式、因果机制、评价标准、实体历史、制度背景、通用定义或约束框架。
- 推理阶段必须回到原始问题，不能停留在泛泛原则。
- 如果任务依赖事实或实时信息，后退问题可用于检索更稳定、更全面的背景；最终答案必须优先服从原始问题和用户给出的上下文。
- 对数学、代码、法律、医学、金融等高风险或可验证任务，要求检查原则适用条件、单位、边界和证据来源。
- 避免要求模型暴露隐藏思维链；需要可解释性时，让模型输出简明、可公开的抽象依据和推理摘要。

## 单轮与两阶段

默认使用单轮模板：一个 Prompt 同时完成后退抽象、抽象层回答、回到原题推理和最终答案。适合普通聊天和用户只想拿到一个可复用提示词的场景。

当脚本或外部工作流可以多次调用模型时，使用两阶段模板：

1. 用抽象阶段 Prompt 生成后退问题，以及相关原则、概念或事实范围。
2. 可选：用后退问题进行检索，把检索结果作为补充上下文。
3. 用推理阶段 Prompt 输入原始问题、后退问题、抽象答案/原则和可选检索上下文。
4. 要求模型基于抽象层回到原题，输出符合用户格式的最终答案。

推荐 JSON 输入：

```json
{
  "task": "回答这道复杂问题",
  "role": "你是一名擅长抽象原则和稳健推理的助手",
  "target_input": "{用户输入的问题}",
  "output_format": "先输出后退问题和抽象依据，再输出最终答案。",
  "constraints": ["最终答案必须回到原始问题", "不要添加无关解释"],
  "trace_visibility": "展示后退问题、抽象依据和简明推理摘要",
  "examples": [
    {
      "original": "具体问题",
      "stepback_question": "更高层、更通用的问题",
      "abstraction": "相关原则、概念、背景事实或通用框架",
      "answer": "如何基于抽象回到原题得到答案"
    }
  ]
}
```

`examples` 可省略。`mode` 为 `single-pass` 时生成完整后退提示；`abstraction` 只生成后退问题和抽象依据 Prompt；`reasoning` 生成基于抽象依据回答原题的 Prompt。

## 注意事项

- 后退提示适合“细节很多导致直接推理容易犯错”的任务；如果任务已经很简单，使用零样本提示即可。
- 后退问题不是改写原题，也不是把原题拆成多个子问题；它应提升到更高语义层。
- 抽象不等于模糊。抽象层要精确、可用于推理，例如公式、原则、标准、历史列表、定义或因果关系。
- 如果后退问题生成得太宽泛，最终答案会漂移；应收紧到“足以覆盖原题所需信息”的最小通用层。
- 如果后退问题生成得太窄，它只是原题复述；应继续上升一层，寻找更稳定的原则或事实范围。
